# RTU 超时与串口移植说明

## UART 必须提供什么

最小严格接口只有两类信息：

1. 每个字节和该字符接收完成时的 32 位时间戳；
2. 来自同一自由运行时钟的当前时间。

时间源可以是 DWT、通用定时器或其他单调回绕计数器。协议库只使用 tick 差值，不要求微秒单位。

时间源和“事件采样方式”是两件事。若 RXNE ISR 读取 DWT/定时器，得到的是
字符完成后的软件观察时刻，时间差会叠加两次 ISR 延迟之差。只有实测最坏
抖动并证明它不会跨越 T1.5 判定边界后，才可把该接法用于严格合规声明。
硬件捕获、UART 接收超时事件或其他与接收硬件关联的边界证据更适合需要
严格证明的产品；这不要求协议库绑定某一个硬件定时器。

UART 的 parity/framing/noise/overrun 也必须作为错误事件上报，并调用 `ld_modbus_rtu_framer_on_error()`。只清硬件标志而不通知接收器，会让已经丢字节的流继续被当作候选帧。

## 每字符位数

必须按真实 UART 格式传入：

| UART 格式 | bits/character |
| --- | ---: |
| 8N1 | 10 |
| 8E1 / 8O1 | 11 |
| 8N2 | 11 |

波特率改变时，应先离线校验新时序；UART 重初始化期间只屏蔽对应 UART 接收路径。硬件配置成功、接收中断恢复之前，再用一个极短临界区执行 `ld_modbus_rtu_framer_reconfigure()`。不要把耗时的 HAL 初始化包在全局 PRIMASK 临界区中，失败时还应回滚原波特率。

## 为什么 poll 等待 T3.5 + 1 字符

设最后字符在 `t0` 完成，下一字符可能在 `t0 + gap` 开始，并在一个字符时间后完成。只有完成后，普通 RXNE 中断才能看到它。

如果任务在 `t0 + T3.5` 立即提交，而一个字符已经开始但尚未完成，协议层会错误认为线路持续静默。等待额外一个字符时间，可以让所有在 T3.5 前开始的字符先到达字节状态机。

这只是接收实现的确认延迟；Modbus 线上合法帧间隔仍然是 T3.5。

## RTOS 并发

推荐把 UART RXNE 作为生产者，把一个协议任务作为消费者：

- ISR：`on_byte()`；
- 任务每 1 ms 左右：短暂屏蔽 UART RX，确认硬件没有 BUSY、RXNE 或接收
  错误 pending 后再调用 `poll()`；若仍有活动，本轮跳过；
- 任务短暂屏蔽 UART RX，执行 `claim()`；
- 恢复中断并处理完整 ADU；
- 再次短暂屏蔽 UART RX，执行 `release()`。

不要在 claim 到 release 的整个协议处理期间保持 PRIMASK，也不要在 ISR 里 memcpy 整个 RTU ADU。

这项硬件静止检查不能省略。若字符在任务关中断后完成，RXNE 可能等待到
临界区结束才进入字节回调；此时先 `poll()` 会错误提交旧帧。检查必须和
`poll()` 位于同一个短串行区，并同时覆盖“字符仍在接收”和“事件已 pending”。

## 不能严格检测的接法

以下输入不足以严格判断块内 T1.5：

```text
ReceiveToIdle DMA callback(data, length, one_end_timestamp)
```

它可以用于明确由硬件边界分割的数据，但不能从一个块末时间戳反推出每两个字节之间是否有非法间隔。移植文档必须说明这一能力边界。

## 建议验证

- 9600、19200、115200；
- 8N1 和至少一种 11 bits/character 格式；
- T1.5 等于边界和大于边界 1 tick；
- T3.5 等于边界；
- 字符在 T3.5 前开始、在 T3.5 后完成；
- 32 位计数器回绕；
- 257 字节超长流；
- ready 帧长时间 claim 时的新帧背压；
- 波特率切换期间无接收字节。
